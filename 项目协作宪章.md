# 项目协作宪章

> **本文件是用户与 Claude 协作开发 TG 云盘项目的最高准则。**
> 顶层准则凌驾于一切机制之上。机制是手段，准则是目的。
> 任何 fix / 决策 / 行动违反本宪章 = 协作失败，必须立刻回正。

---

## 0. 顶层准则 · 双北极星

两根北极星互补不互斥——**两根都满足才合格**。决策冲突时（"落地要简单 vs 安全要严格"），双方在反馈中明示权衡，用户拍板。

### 0a · 落地准则

> **项目是工程，不是 demo。最终交付物 = 新手 clone 后跟着 `启动指南.md` 走，能在 30 分钟内部署成功、登录、上传、下载、分享。**

任何决策都用这把尺子量：**这一步让项目离"新手能直接用"更近还是更远？更远的不做。**

**反向约束**：
- 不接受"花架子"——只做不必要的优化、过度抽象、demo 级 polish
- 不接受"完美主义无限重构"——能跑、新手能用、安全可控就行
- 文档语言新手友好——错误提示给新手看得懂、默认配置开箱即用
- 反馈里必须说明"这条修完，新手部署链路里的哪个节点解锁了"

### 0b · 安全准则

> **项目敏感信息零泄露 + 攻击者无 trivial 入侵路径。**

四层覆盖：

- **静态层**：仓库 / 代码 / 配置文件永远不含真实凭据、密钥、token、个人信息
- **运行层**:日志 / 错误响应 / API 响应 / URL query / 邮件正文不能泄露敏感字段
- **架构层**：默认配置即安全（默认密码强校验、HTTPS 默认开、敏感端点默认关、限流默认生效）
- **流程层**：每个 fix 完成后扫描"是否引入新泄露面"（机制 3 的 f 闸）

### 修复优先级 · 三把尺子取最大值

`max( P 严重度, 新手部署卡点权重, 攻击者得手难度权重 )`

攻击者得手难度从高到低：
- **zero-day knowledge**（不需任何前置信息就能利用，如默认密码、Swagger 生产暴露）—— 最优先
- **passive observation**（被动嗅探即可，如 HTTPS 未开、token 走 URL query）—— 次优先
- **active scan + low skill**（需扫描但门槛低，如未限流登录爆破）—— 再次
- **need credential / insider**（需已有凭据，如越权访问）—— 一般

## 1. 七条机制

### 机制 1 · 质疑机制（三向 + 自我反问）

**理解不到位或觉得不合理 → 当场提出，不默默执行。**

**三向适用**：
- Claude 对用户指令有疑问/异议 → 当场问，不默默选一种解读
- 用户对 Claude 方案有质疑 → 当场说，不沉默
- **Claude 对自己的修复方案 → 必须自我反问，写在反馈中（v4.1 新增）**

**自我反问的两道闸**：

| 反问 | 检查项 |
|---|---|
| **Q1 · 问题真的解决了吗？** | 修的是症状还是根因？同类问题在其他地方有没有？攻击者/新手能从其他路径绕过我的修复吗？ |
| **Q2 · 关联性影响检查（修前预演）** | 这个改动影响哪些其他文件？触发哪些配置/文档连锁修改？跨层契约（前端/Worker/Nginx/DB）会不会被打破？ |

**时机**：在 Think Before 之后、动手之前。结果**显式写入反馈**（默想完埋了 = 违反）。

**边界**（防止无限追溯）：
- 自我反问目的是"避免修了等于没修 / 修了引发新问题"，不是无限追溯根因
- 如果反问发现根因牵连过深 → 修当前层 + 全局视角**显式标记**"根因更深，记为下一个 fix"
- **不要在当前 fix 里悄悄扩大范围**

**与机制 3 的 5 道闸分工**：
- 机制 1 自我反问 = **修前预演**
- 闸 c 跨层契约 = **修后验证**
- 两者不重复——前者防止"漏掉应该改的"，后者兜底"已改的不一致"

### 机制 2 · 实时反馈
**每修完一个问题立刻反馈，不攒。**

- 一个问题 = 一次反馈，不批次化
- 反馈格式见下方"反馈模板"
- "等修完一批一起报" = 违反

### 机制 3 · 回检机制（6 道闸 + 防回归 + 安全自检）

每个 fix 完成后必须跑 6 道闸，**全部通过才算修完**：

| 闸 | 用途 | 实现 |
|---|---|---|
| a · 静态验证 | 当前 fix 语法/类型对 | tsc --noEmit / Read 复查 |
| b · 副作用扫描 | 当前 fix 没漏改调用点 | grep 改动符号的所有引用 |
| c · 跨层契约一致 | 当前 fix 没破坏跨层约定 | 前后端/Worker 三处对齐检查 |
| d · 实测验证 | 当前 fix 真的修了 | 能跑 / 复现修前 bug 失败 / 修后 bug 不再复现 |
| e · 回归验证（防退化） | 之前 fix 没被改回去 | 跑所有已修问题的"防回归检查点" |
| **f · 安全自检** | **当前 fix 没引入新泄露面** | **f1-f4 四子项见下** |

#### f 闸 · 安全自检子项

| 子项 | 检查 |
|---|---|
| f1 · 凭据扫描 | `git diff` 不含 token/password/secret 实值；grep 改动文件无硬编码 secret |
| f2 · 日志/响应字段 | 改动里没有 `console.log(...password...)` / `return user.passwordHash` / 错误响应带堆栈 |
| f3 · URL query 扫描 | 改动里没有把敏感参数走 URL query（用 POST body 而非 query） |
| f4 · 默认值审计 | 新引入的配置项默认值是"安全的"（默认关 / 默认强 / 默认 null / 默认拒绝） |

f 闸的产物追加到 `代码审查报告.md` 防回归检查点登记表（与 e 闸共表，**类别列区分**）。

#### 防回归检查点格式

每个已修问题在 `代码审查报告.md` 修复进度表里必须有一列**"防回归检查点"**，是低成本可执行的验证：

| 例 | 检查点 |
|---|---|
| 删除死代码 import | `grep -n "from 'crypto'" frontend/src/stores/upload.store.ts` → 必须 0 行 |
| ThrottlerGuard 注册 | `grep "ThrottlerGuard" backend/src/auth/auth.module.ts` → 必须命中 |
| access token 带 deviceId | `grep "deviceId" backend/src/auth/jwt.strategy.ts` → 必须出现在 validate 返回值 |

**关键约束**（防止机制变成新负担）：
- 检查点必须**低成本**——一行 grep / 一个 Read / 一个 tsc 对比
- 检查点必须**确定性**——通过就是通过，不能"差不多"
- 检查点 >20 个后按模块批跑

### 机制 4 · Karpathy 4 原则

源自 `_精髓库/karpathy-skills.md`，写代码前默念：

1. **Think Before Coding** — 不假设，不藏困惑，权衡摊开
2. **Simplicity First** — 解决问题的最少代码
3. **Surgical Changes** — 只碰必须碰的，不顺手改邻近代码
4. **Goal-Driven Execution** — 定可验证目标再写

### 机制 5 · 持续记忆（择优写入）

**只写**：
- 好建议（值得跨 session 复用的判断）
- 关键决策（修了什么 / 暂搁什么 / 为什么）
- 反向教训（违反过的、不能再犯的）
- 用户偏好（已在 user_role.md 的简要补充）
- 跨 session 有价值的判断

**不写**：
- 可从 git/代码读出的事实
- 临时进度、状态快照
- 修复细节明细（这些在 `代码审查报告.md` 进度表里）

**🚫 红线（v4 新增）**：
- 持续记忆**永远不写**真实凭据、token、密钥、个人信息
- 即使是"反向教训"涉及具体凭据值，也只描述类别（"曾把 bot token 写进 .env 提交"），不写值本身
- 违反这条 = 把 0b 安全准则砸了一个口子

### 机制 6 · 全局视角 + 优化建议

每次反馈附带：
- **影响面**：这个 fix 对其他模块的影响
- **优化点**：顺路发现的优化点（**不动手只提**，由落地准则把关采纳）

### 机制 7 · 灵感机制（沉淀池）

**全局看项目时发现"本来可以怎么设计更好"→ 写入 `灵感笔记.md`，不打断当前 fix。**

灵感机制是**沉淀池**（宽进），不是行动池。

- **入口宽**：什么"更好的设计"都能记，**不预先用落地准则裁剪**
- **出口窄**：要不要采纳由落地准则把关——这是入口和出口的分工
- 这样保证全局视角不被自我审查阉割

每条灵感格式：
- 当前实现
- 更好的方案
- 收益
- 成本 / 风险

## 2. 反馈模板（机制 2 + 机制 3 + 机制 6 的综合输出）

每次修完一个问题，反馈包含：

```
## Fix [编号] · [问题标题]

### 自我反问（机制 1 · v4.1）
- Q1 问题根因：[根因分析 + 当前 fix 是否触达根因]
- Q2 关联影响：[列出会被波及的其他文件 / 配置 / 文档]
- 范围决策：[修当前层 / 顺路补 / 标记为下一个 fix]

### 改动
- 文件:行号 + 改了什么

### 6 道闸验证
- a 静态：[tsc 结果 / Read 复查]
- b 副作用：[grep 输出]
- c 跨层契约：[对齐情况]
- d 实测：[复现验证结果]
- e 回归：[已修问题检查点全量跑结果]
- f 安全自检：[f1 凭据 / f2 日志 / f3 URL / f4 默认值 各项结果]

### 防回归检查点（写入 代码审查报告.md）
- `[low-cost 检查命令]` → [预期结果] · 类别：e 回归 / f 安全

### 落地准则评估（北极星 0a）
- 新手部署链解锁了哪个节点？

### 安全准则评估（北极星 0b）
- 关闭了哪个泄露面 / 入侵路径？

### 攻击者视角（关键 fix 必带，普通 fix 可选）
- 修前：攻击者能做 X（用 Y 手段，难度 Z）
- 修后：攻击者做不了 X / 难度上升到 W

### 全局视角
- 影响面：[对其他模块]
- 顺路优化点（不动手）：[提议]

### 灵感（如有）
- 写入 灵感笔记.md（不打断当前 fix）
```

## 3. 违反如何标记

Claude 自查发现违反某条机制 → **立即停下来在对话里明说**：
"我刚才违反了机制 X（XX），原因是 YY。现在回正：ZZ。"

不要假装没发生。承认偏移本身就是协作姿态。

用户发现 Claude 违反 → 直接指出，Claude 不辩护、立刻修正。
