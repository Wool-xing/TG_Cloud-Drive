# 灵感笔记

> **用途**：全局视角看 TG 云盘项目时发现"本来可以怎么设计更好"的沉淀池。
>
> **机制**：宽进窄出。入口宽——什么"更好的设计"都能记；出口窄——要不要采纳由 `项目协作宪章.md` 的落地准则把关。
>
> **不打断当前 fix**——灵感是沉淀，不是行动。
>
> 详见 `项目协作宪章.md` 机制 7。

---

## 格式

每条灵感按以下格式：

```
### #N · [一句话标题]
- **日期**：YYYY-MM-DD
- **触发**：在做什么时想到的
- **当前实现**：现在项目是怎么做的
- **更好的方案**:[描述]
- **收益**：[安全 / 性能 / 可维护 / 新手友好 / ...]
- **成本/风险**：[实施成本、引入的复杂度、对部署的影响]
- **采纳判断**（待用户决策）：[落地准则视角下是否值得采纳]
```

## 灵感列表

### #1 · 统一 `make doctor` / `make check-regressions` 入口
- **日期**：2026-05-13
- **触发**：B1 防回归检查点登记落地，发现已有 10 个低成本命令散落在文档里
- **当前实现**：检查点用 `grep` / `test` 命令散在 `代码审查报告.md` 表格中，每次回归验证靠人工 / Claude 逐条粘贴
- **更好的方案**：Makefile 新增两个 target：
  - `make doctor` —— 启动前自检（检查 `.env` 不含 `CHANGE_ME_*` 占位、Node 版本、Docker 可用性、JWT_SECRET 长度 ≥32）。新手部署痛点之一是"配错了启动失败但报错不清晰"，doctor 给出明确诊断
  - `make check-regressions` —— 全量跑 `代码审查报告.md` 里的所有防回归检查点，输出通过/失败统计
- **收益**：
  - 新手友好（落地准则）：一行命令拿到部署前体检结果
  - 回归验证自动化：Claude 修完一个 fix 跑 `make check-regressions` 比逐条粘贴快
  - 与 P1-I13（启动指南强制改密）天然契合——doctor 是它的执行体
- **成本/风险**：
  - 成本：Makefile 不到 30 行，但每加一个新 fix 都要往 `check-regressions` 里加 1-3 行（自动化负担微小）
  - 风险：Makefile 用 bash 语法，Windows 用户需要 git bash / WSL（项目当前已用 Makefile，所以风险已经存在）
- **采纳判断**（待用户决策）：**强烈推荐**——直接服务落地准则（新手友好）+ 强化机制 3（回归验证可执行化）。建议在修完 P1-I13 时一并实施
