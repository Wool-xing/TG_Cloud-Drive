# 灵感笔记

> **用途**：全局视角看 TG 云盘项目时发现"本来可以怎么设计更好"的沉淀池。
>
> **机制**：宽进窄出。入口宽——什么"更好的设计"都能记；出口窄——要不要采纳由 `项目协作宪章.md` 的落地准则把关。
>
> **不打断当前 fix**——灵感是沉淀，不是行动。
>
> 详见 `项目协作宪章.md` 机制 7。

## 格式

每条灵感按以下格式：

```
### #N · [一句话标题]
- **日期**：YYYY-MM-DD
- **触发**：在做什么时想到的
- **当前实现**：现在项目是怎么做的
- **更好的方案**:[描述]
- **收益**：[安全 / 性能 / 可维护 / 新手友好 / ...]
- **成本/风险**：[实施成本、引入的复杂度、对部署的影响]
- **采纳判断**（待用户决策）：[落地准则视角下是否值得采纳]
```

## 灵感列表

### #1 · 统一 `make doctor` / `make check-regressions` 入口
- **日期**：2026-05-13
- **触发**：B1 防回归检查点登记落地，发现已有 10 个低成本命令散落在文档里
- **当前实现**：检查点用 `grep` / `test` 命令散在 `代码审查报告.md` 表格中，每次回归验证靠人工 / Claude 逐条粘贴
- **更好的方案**：Makefile 新增两个 target：
  - `make doctor` —— 启动前自检（检查 `.env` 不含 `CHANGE_ME_*` 占位、Node 版本、Docker 可用性、JWT_SECRET 长度 ≥32）。新手部署痛点之一是"配错了启动失败但报错不清晰"，doctor 给出明确诊断
  - `make check-regressions` —— 全量跑 `代码审查报告.md` 里的所有防回归检查点，输出通过/失败统计
- **收益**：
  - 新手友好（落地准则）：一行命令拿到部署前体检结果
  - 回归验证自动化：Claude 修完一个 fix 跑 `make check-regressions` 比逐条粘贴快
  - 与 P1-I13（启动指南强制改密）天然契合——doctor 是它的执行体
- **成本/风险**：
  - 成本：Makefile 不到 30 行，但每加一个新 fix 都要往 `check-regressions` 里加 1-3 行（自动化负担微小）
  - 风险：Makefile 用 bash 语法，Windows 用户需要 git bash / WSL（项目当前已用 Makefile，所以风险已经存在）
- **采纳判断**（待用户决策）：**强烈推荐**——直接服务落地准则（新手友好）+ 强化机制 3（回归验证可执行化）。建议在修完 P1-I13 时一并实施

### #2 · 代码层全绿 ≠ make quickstart 能跑（反向教训）
- **日期**：2026-05-14
- **触发**：用户实测 `git clone https://github.com/Wool-xing/TG_Cloud-Drive.git tg-pan && cd tg-pan && make quickstart` 跑不通，立刻揭出 6 个端到端必炸点（旧 218 条审查 + 19 个已修 fix + 91 项防回归检查点 **0 条覆盖**）
- **当前实现**：6 个并行专项 agent 审了代码层（auth / 业务 / 基础设施 / 前端页面 / 前端组件 / 全局架构），主审 cross-validate，机制 3 闸 d "实测" 普遍只做静态推理 ("理论应通即 OK")，没人模拟新手端到端流程
- **实际暴露的 6 个 quickstart P0 炸点**（旧报告 0 覆盖）：
  1. `docker-compose.yml:74` `environment.NODE_ENV: production` 覆盖 .env 写的 development（compose 规则：environment > env_file）
  2. `quickstart.sh:245 + Makefile:70 + 启动指南.md:78` seed 用 `npx ts-node src/database/seed.ts`，但 production image 既无 ts-node 也无 src/
  3. `README.md:14` `git clone <this-repo-url>` 占位符 → bash 当 redirect 报错
  4. `quickstart.sh:214 + Makefile certs-dev` 用 `docker run alpine/openssl`，国内/受限网络 image pull 失败
  5. backend/frontend `Dockerfile` 用 `npm install` 而非 `npm ci`，且未 COPY package-lock.json → 镜像不可重复构建 + 无 .dockerignore
  6. `env-validator.ts WEAK_DEFAULTS` 含 `'admin'`，quickstart 写 `ADMIN_USERNAME=admin` 被 isPlaceholder 拒 → backend exit(1)
- **共同根因**：6 agent 都跑代码层 grep/tsc 审查，闸 d "实测"被解释为"静态推理通过即 OK"。没人**模拟新手端到端流程**（build → up → bootstrap → seed → login）。**落地北极星 0a 漏检**
- **更好的方案**：宪章机制 3 闸 d 升级（建议 v4.2），任何改以下文件的 fix，闸 d 必须包含**三层落地实测**：
  1. **静态终值实测** — `docker compose config` 解析查终值（NODE_ENV / DATABASE_URL / REDIS_URL / 8 个 required env 不被意外覆盖）
  2. **静态全链路走读** — Dockerfile COPY 覆盖 → npm 装包 → image 内 dist/ 存在性 → seed/migration 入口可达 → entities 类引用都编译
  3. **真跑一次端到端** — 用户/CI 跑 quickstart 到管理员登录（前 2 层抓静态可见炸点，第 3 层兜底）
- **触发文件清单**：`docker-compose.yml` / `Dockerfile` / `nginx.conf` / `Makefile` / `quickstart.sh` / `env-validator.ts` / `.env.example` / `seed.ts` 任意改动
- **不能用代替端到端验证的**："6 agent 全说 OK" / "tsc 全绿" / "91 项防回归检查点全绿" / "代码层 P0 修完了"
- **收益**：
  - 落地北极星 0a 真正生效（不是话术）
  - 节约用户实测时间（用户跑 5 分钟发现 6 个炸点 = 我们应该提前抓出）
  - 闸 d 不再是橡皮图章
- **成本/风险**：
  - 每次部署链路 fix 多花 10-15 分钟做"静态终值实测"（docker compose config + 链路走读）
  - 真跑端到端要 docker + Telegram 凭据，开发者本地有，但 Claude 实测不易（依赖宿主环境）
  - **关键**：这就是为什么需要**用户/CI 配合**——Claude 做前 2 层静态，用户做第 3 层实测
- **采纳判断**：**强烈推荐**，建议升级宪章 v4.2 加 `机制 3 闸 d 三层细化` + 加 `反向教训第 X 条 · 代码全绿 ≠ 落地能跑`。本灵感本身就是从落地准则出口窄过的实战教训，应当被采纳。
