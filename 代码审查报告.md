## 一、一句话结论

**当前代码不是"差几个 bug"，而是"从未跑通过完整加密上传→换设备→下载链路"的早期 Demo**。骨架对、原语对，但每个安全/数据完整性环节上都有至少一处断点。

**P0 共 14 类，不修不能上线。**

## 二、全局结构性诊断

项目的"骨架"看起来完整——entity 设计合理、端点齐全、E2E 加密 + 私密空间 + 分享 + 文件锁都有专门字段——但执行层全是裂缝：

### 1. 承诺 > 实现（最严重）
- 宣传 E2E 加密但**密码出浏览器**（后端 bcrypt 校验）
- DEK 进 URL fragment 永久泄露
- IV 不存导致多分片文件 100% 解密失败
- 明文上传分支（MEK 缺失时）静默存在
- 用户会以为数据是端到端加密的，实际有多条服务端可读路径

### 2. 没有事务边界
node + chunk + key + user.usedBytes + share + telegram 横跨多表多服务，全靠"祈祷成功"。任何中间步骤失败留垃圾数据。

### 3. 没有可观测性
`catch(() => {})` 漫天飞 + audit log 写失败被吞 + 无 trace-id 串联。出问题后无法溯源。

### 4. 信任边界不清
浏览器 → Worker → Nginx → Backend → Telegram 五跳，每跳都假设"上一跳已经校验"，但实际每跳都能被绕过。

### 证据链：作者从未跑通完整链路
- 多分片加密 IV 没存——多 chunk 加密文件从未成功下载过
- ThrottlerGuard 注解满天飞但 Guard 没注册——只要打开过 DevTools 看一次"登录失败 100 次还能再试"就会发现
- `import { v4 as uuid } from 'crypto'`——npm run build 不会过

## 三、🔴 P0 致命问题清单（14 类）

### A. 加密承诺彻底破产

#### A1. AES-GCM 同 DEK 跨分片 IV 复用 — 密码学级灾难
- **文件**：
  - `frontend/src/utils/crypto.ts:44` 每 chunk 生成独立 IV
  - `frontend/src/stores/upload.store.ts:135` 把所有 chunk 的 IV 都丢进同一个 `iv` form 字段
  - `backend/src/files/entities/file-chunk.entity.ts` **完全没有 iv 字段**
  - `backend/src/files/files.service.ts:67-113` `uploadChunk` 只在首块时把 iv 存入 `NodeKey`
  - `frontend/src/components/preview/PreviewModal.tsx:78-84` 用 `NodeKey.iv` 解所有 chunk，代码注释自承 "server must provide per-chunk IVs"
- **风险**：(1) 单 chunk 文件勉强能跑；(2) 多 chunk 文件 100% 解密失败；(3) 若 GCM nonce 复用被攻击者捕获两份密文，可用 forbidden attack 恢复明文 + 伪造任意密文
- **修复**：FileChunk 加 iv 列 + 后端 uploadChunk 存 chunk iv + 前端按 chunk iv 解密（三处必须同步修）

#### A2. 上传无 MEK 时静默走明文
- **文件**：`frontend/src/stores/upload.store.ts:86`
- **问题**：`if (mek) {加密} else {chunkData = buffer}` — 用户刷新页面后 MEK 在内存丢失但 zustand 仍说"已登录"，于是无声切到明文上传
- **风险**：端到端加密承诺被悄悄违反，用户不知道
- **修复**：无 MEK 时禁止上传，强制弹窗要求重新输入密码派生

#### A3. MEK 派生用登录密码 = "端到端加密"是营销话术
- **文件**：架构级，涉及 `crypto.ts` + 后端登录链路
- **问题**：登录密码必须发后端 bcrypt 校验，攻击者拿到后端=旁路一切加密。文档却宣称"管理员也无法查看用户文件"
- **修复**：要么改文档（"后端不持久化明文"），要么引入 SRP/OPAQUE 让密码真的不出浏览器
- **状态**：用户决策"暂不动，润下次决策"

#### A4. 分享 DEK 嵌在 URL fragment 中、无吊销、永久泄露
- **文件**：`frontend/src/utils/crypto.ts:95-104` `exportDEKAsBase64`
- **问题**：把 DEK base64 放进 `#fragment`，链接被截图/转发/进入浏览器历史，DEK 永久外泄
- **修复**：分享条目存独立"share DEK"（用 owner MEK 包裹的二级密钥），可数据库侧 revoke

### C. 信任边界与基础设施

#### C1. HTTPS 完全没启用
- **文件**：`nginx.conf:166-189` HTTPS server 块整段注释；`启动指南.md:50` 引导用户 `http://localhost`
- **风险**：登录密码（= MEK 派生原料）走明文 = 同 Wi-Fi 抓包即可解密用户全部文件
- **修复**：默认启用 HTTPS server 块；HTTP→HTTPS 301 重定向；HSTS max-age ≥ 6 个月

#### C2. CORS 配置在生产环境放过任意 localhost
- **文件**：`backend/src/main.ts:22` `origin: [frontendUrl, /localhost:\d+$/]`
- **问题**：正则匹配 `evil.com.localhost:1234`，配合 `credentials: true` 等价 CSRF 防线被打穿
- **修复**：生产模式下移除 localhost 正则，仅 `[frontendUrl]`

#### C3. Swagger 生产暴露
- **文件**：`backend/src/main.ts:34-40`
- **问题**：无 NODE_ENV 判断，`/api/docs` 完整 API 地图直接公开
- **修复**：`if (process.env.NODE_ENV !== 'production')` 包裹

#### C4. Cloudflare Worker 是"通用 Telegram API 透传"
- **文件**：`worker/src/index.ts:260-312` `/api/tg/:method`
- **问题**：method 由 URL 拼接、无白名单、`chat_id` 接受请求体覆盖
- **风险**：secret 一旦泄露 = bot 完全失陷、可向任意 chat 发垃圾、可 `getUpdates` 读所有对话
- **修复**：method 白名单（仅 sendDocument/getFile/deleteMessage）；chat_id 由 Worker 强制赋值不接受覆盖

#### C5. Worker CORS = `*`，配合下载 token = 任意网站可下你的文件
- **文件**：`worker/src/index.ts:28-33`
- **修复**：改成业务前端域名白名单；对 `/file/:token` 增加 Referer 校验

#### C6. 后端 buildSignedDownloadUrl 与 Worker createFileToken 算法不兼容
- **文件**：
  - 后端 `backend/src/telegram/telegram.service.ts:79-87` `'${fileId}:${exp}'` + 截断 16 hex（64 bit）
  - Worker `worker/src/index.ts:100-124` `'${fileId}.${exp}'` + 完整 64 hex
- **风险**：一旦后端接 Worker 下载链路全部 403；16 hex 签名暴力可破
- **修复**：算法统一（推荐用 Worker 版本完整 HMAC）；签名长度 ≥ 32 hex；用独立 FILE_SIGNING_KEY 不复用 JWT_SECRET

### D. 业务核心崩坏

#### D1. 分享 maxDownloads 可被无限绕过
- **文件**：`backend/src/shares/shares.controller.ts:69-77`、`shares.service.ts:181-183`
- **问题**：`recordDownload` 客户端可选调用，独立于真正下载链路；`accessShare` 检查 downloadCount 在累加前
- **风险**：攻击者不调 `recordDownload` 就能无限下；maxDownloads=1 形同虚设
- **修复**：真正下发文件分片下载链接的接口才做计数与上限校验；用 SQL 条件 update 原子递增

#### D2. 分享密码无任何限流
- **文件**：`backend/src/shares/shares.service.ts:131-194`
- **问题**：Public 端点，错误密码只抛 401，4-6 位数字密码毫秒级爆破
- **修复**：按 token + IP 维度在 Redis 做错误次数限流（5 次/分钟，30 次/小时）

#### D3. uploadChunk 无配额检查
- **文件**：`backend/src/files/files.service.ts:67-113`
- **问题**：无 `usedBytes + 本次 ≤ quotaBytes` 校验
- **风险**：用户可无限上传打爆 TG 频道
- **修复**：首块拿到客户端声明 totalSize 先校验；最后落地再校验防 TOCTOU

#### D4. permanentDelete 不递归子节点 + 不清 Telegram 残留 + 不清 share 缓存
- **文件**：`backend/src/files/files.service.ts:193-204`
- **问题**：
  - 没限制 `deletedAt: Not(IsNull())`，可绕过回收站直接 hard delete 活跃文件
  - 文件夹的 children 没有递归处理，子文件夹下的 chunks 不会被删 TG
  - Share token 的 Redis 缓存不会失效
- **修复**：加 deletedAt 校验；递归收集所有后代节点和 chunks 先调 deleteMessage 再 DB delete；主动失效 share 缓存；包事务

#### D5. 登录用户全表扫描 + 逐条解密
- **文件**：`backend/src/auth/auth.service.ts:165-178` `findUserByIdentifier`
- **问题**：邮箱/手机登录加载全表 user → 逐条 AES-GCM 解密。emailEncrypted 用随机 IV，无法索引匹配
- **风险**：万级用户时单次登录数百 ms；几十并发即可拖垮节点（DoS）；存在计时旁路
- **修复**：加 `emailHash = HMAC-SHA256(masterKey, normalize(email))` 列 + 唯一索引

#### D6. 修改密码后不撤旧 session
- **文件**：`backend/src/users/users.service.ts:104-110` `changePassword`
- **问题**：注释假装"Invalidate all sessions"实际什么都没做
- **风险**：密码改了，攻击者持有的 access/refresh token 仍 30 天有效
- **修复**：删除所有 Device 记录 + Redis `force_logout:` 标记

#### D7. 强制下线在 Redis 失败时 fail-open
- **文件**：`backend/src/auth/jwt.strategy.ts:25-33`
- **问题**：注释明确写 "Redis unavailable — continue (fail open)"
- **风险**：Redis 抖动 = 被强制下线的 token 重新可用；admin 封号失效
- **修复**：Redis 失败 fail-close 抛 503 或 401

#### D8. 全局节流形同虚设
- **文件**：`backend/src/app.module.ts:34`、`backend/src/verification/verification.controller.ts:20`
- **问题**：`ThrottlerModule.forRoot` 注册了但 **`ThrottlerGuard` 从未注册为 APP_GUARD**
- **风险**：`@Throttle()` 装饰器是装饰品；登录/注册/refresh/验证码全部裸跑
- **状态**：✅ [已修复] B1 修复，在 AuthModule 注册了 APP_GUARD + login/register/refresh 加 @Throttle
