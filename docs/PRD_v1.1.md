# TG云盘系统 · 产品需求文档（PRD）

**版本：** v1.1  
**日期：** 2026-05-02  
**定位：** 基于 Telegram 存储的个人/团队云盘，国内无需 VPN 直接访问

---

## 一、项目概述

### 1.1 产品定位

以 Telegram 作为底层存储介质，构建一套对标百度网盘的云盘系统。用户通过 Web 客户端操作，文件实际存储于 Telegram 频道，国内用户无需 VPN 即可完成上传、下载、管理全流程。

### 1.2 核心优势

| 优势 | 说明 |
|------|------|
| 无限存储 | 基于 Telegram 无限存储特性 |
| 国内直连 | 通过 Cloudflare Workers 代理，无需 VPN，无需备案服务器 |
| 低成本 | 零存储费用，Cloudflare Workers 免费额度覆盖中小用户量 |
| 端到端加密 | 文件上传前客户端加密，服务端无法读取原始内容 |
| 高安全 | 多层加密 + 隐私空间 + RBAC 权限模型 |

### 1.3 Telegram 配置参数

> ⚠️ 以下参数必须加密存储于服务端环境变量，禁止明文写入代码或提交至代码仓库

```
Bot 名称：Woolro_Bot
Bot Token：存于环境变量 TG_BOT_TOKEN（AES-256 加密）
频道 ID：存于环境变量 TG_CHANNEL_ID（AES-256 加密）
```

## 二、国内直连架构（无需国内服务器）

### 2.1 整体方案：Cloudflare Workers 代理

无国内服务器的情况下，借助 Cloudflare 全球 CDN 节点实现国内直连：

```
用户（国内浏览器）
    │
    ▼
Cloudflare Workers（workers.dev 域名或自定义域名，Cloudflare 在国内有直连节点）
    │
    ├─ 代理 Telegram Bot API 上传请求 ──► Telegram 服务器
    │
    └─ 代理文件下载请求 ──────────────► Telegram CDN ──► 返回文件流
```

### 2.2 Cloudflare Workers 职责

| Workers 路由 | 职责 |
|-------------|------|
| `/api/tg/*` | 代理 Telegram Bot API 调用（上传、获取文件） |
| `/file/{token}` | 代理文件下载，校验 token 合法性后流式返回文件 |
| `/upload-chunk` | 接收分片数据并转发至 Telegram |

### 2.3 下载外链格式

```
https://cdn.yourdomain.com/file/{signed_token}
```

- `signed_token`：包含 file_id + 过期时间 + HMAC 签名，防篡改防盗链
- Cloudflare Workers 验证签名后，向 Telegram CDN 获取文件并流式返回
- 支持 Range 请求（视频在线播放 seek 操作必需）

### 2.4 成本估算与扩容策略

| 阶段 | 日请求量 | 方案 |
|------|----------|------|
| 初期 | < 10 万次/天 | Cloudflare Workers 免费套餐（10 万请求/天） |
| 成长期 | 10 万 ~ 1000 万/天 | Workers Paid（$5/月起） |
| 大规模 | > 1000 万/天 | 增加 HK/海外服务器作为二级缓存节点 |

### 3.2 注册

**注册策略：** 开放公开注册

**注册流程：**

```
填写用户名 + 密码 + 手机号或邮箱
    │
    ▼
发送验证码（v1.0 本地随机生成 6 位数字码，通过邮件/页面展示）
    │
    ▼
验证码校验通过 → 账号创建成功
```

**验证码设计（可扩展架构）：**

```typescript
// 验证码接口抽象层，未来替换为短信/邮件服务只需更换实现
interface VerificationCodeProvider {
  send(target: string, code: string): Promise<void>
}

// v1.0 本地实现：开发环境控制台输出，生产环境写库供前端展示
class LocalVerificationProvider implements VerificationCodeProvider {
  async send(target: string, code: string) {
    await db.verificationCodes.upsert({ target, code, expiresAt: Date.now() + 5 * 60 * 1000 })
  }
}

// 未来扩展占位（实现接口即可热切换）
// class AliyunSmsProvider implements VerificationCodeProvider { ... }
// class SmtpEmailProvider implements VerificationCodeProvider { ... }
```

**注册规则：**
- 用户名：4~20 位，仅允许字母、数字、下划线，不可重复
- 密码：至少 8 位，含大小写字母 + 数字 + 特殊字符
- 手机号或邮箱：二选一，用于找回密码，可后续在个人中心补充
- 验证码：6 位随机数字，有效期 5 分钟，每个目标 1 分钟内只能发送 1 次

### 3.5 密码管理

## 四、文件管理系统

### 4.1 目录结构

- 支持多级目录（层级无上限）
- 每个目录下最多创建子文件夹数：默认 10 个，管理员可全局配置此上限
- 根目录按用户完全隔离（用户只能访问自己的文件）
- 文件夹操作：新建、重命名、移动（含子树）、删除（非空时提示递归删除确认）
- 回收站：删除的文件/文件夹保留 30 天，可还原至原目录或永久删除，支持批量清空

### 4.2 文件上传

#### 上传流程（端到端加密）

```
用户选择文件
    │
    ▼
浏览器端：AES-256-GCM 加密文件内容，生成加密密钥（DEK）
    │
    ▼
文件 MD5 校验（用于秒传判断，基于加密前内容）
    │
    ▼
文件按 20MB 分片（对加密后内容分片）
    │
    ▼
各分片通过 Cloudflare Workers 上传至 Telegram
    │
    ▼
全部分片上传完成，Telegram 返回各分片 file_id
    │
    ▼
后端数据库记录：文件名、大小、分片 file_id 列表、加密后的 DEK、目录位置
    │
    ▼
生成带签名的下载外链
```

#### 加密密钥管理

- 每个文件生成独立的 AES-256-GCM 密钥（DEK，Data Encryption Key）
- DEK 使用用户主密钥（MEK，Master Encryption Key，由用户密码 PBKDF2 派生）加密后存入数据库
- MEK 不存服务端，每次解密需用户登录状态下重新派生
- 用户修改密码时，重新加密所有 DEK（异步后台任务）
- 管理员无法解密用户文件（真正的端到端加密）

#### 上传特性一览

| 特性 | 说明 |
|------|------|
| 分片上传 | 单片 20MB，支持并发上传（最多 3 片同时上传） |
| 断点续传 | 网络中断后从最后成功分片继续，刷新页面后队列恢复 |
| 秒传 | 同一用户相同文件（MD5 相同）跳过重复上传 |
| 批量上传 | 多文件/文件夹拖拽，自动保留文件夹层级结构 |
| 格式无限制 | 任意格式，无类型过滤（可由管理员配置黑名单） |
| 超大文件 | 单文件无上限（分片拼接，Telegram 单消息最大 2GB） |
| 上传队列 | 进度条 + 速度 + 剩余时间，支持暂停/恢复/取消单个任务 |
| 同名冲突 | 提示：覆盖 / 自动重命名（加 _1、_2 后缀）/ 跳过 |
| 上传限速 | 用户可设置最大上传带宽占用 |

### 4.3 文件下载

- 下载时：后端返回签名外链 → 浏览器下载加密文件 → 客户端解密 → 保存到本地
- 支持在线预览（解密后在内存中渲染，不落盘）
- 批量下载：队列逐一解密后打包为 ZIP（浏览器内存流，避免大文件 OOM）
- 下载外链有签名保护，默认有效期 1 小时（用于分享场景可自定义延长）

### 4.4 文件操作

| 操作 | 说明 |
|------|------|
| 重命名 | 文件/文件夹均可，实时验证重名 |
| 移动 | 拖拽目标文件夹，或"移动到"弹窗选择目录树 |
| 复制 | 支持跨目录复制（数据库层复制元数据，不重复占用 Telegram 存储） |
| 删除 | 单个/批量删除，先移入回收站 |
| 收藏 | 加入收藏夹，跨目录快速访问 |
| 分享 | 生成分享链接（见 4.8） |
| 加锁 | 设置访问密码（见 4.6） |
| 属性 | 查看文件名、大小、类型、上传时间、MD5（加密前）|
| 标签 | 自定义标签，支持多标签，按标签检索 |

### 4.5 文件查看与预览

**分类筛选标签：**  
全部 / 文档 / 图片 / 视频 / 音频 / 压缩包 / 其他

**在线预览（解密后在浏览器内存中渲染）：**

| 类型 | 支持格式 |
|------|----------|
| 图片 | JPG、PNG、GIF（动图）、WebP、SVG、HEIC |
| 视频 | MP4、MOV、MKV（H.264/H.265，分片流式解密播放） |
| 音频 | MP3、FLAC、WAV、OGG、AAC |
| 文档 | PDF（分页懒加载）、TXT、Markdown（渲染预览）|
| 代码 | 主流语言语法高亮（highlight.js） |

**视图切换：**
- 列表视图（详细信息）
- 网格视图（缩略图，图片类型优先展示真实缩略图）

### 4.6 文件/文件夹加锁

- 默认无锁，任何文件/文件夹均可设置访问密码
- 加锁后行为：
  - 文件列表中可见（显示锁定图标）
  - 点击后弹出密码输入框，验证通过后才能预览/下载
  - 错误密码尝试超 5 次，冷却 10 分钟
- 支持修改密码、取消锁定
- 父文件夹加锁后，子内容自动受保护（无需逐个加锁）
- 加锁密码独立于账号密码，bcrypt 哈希存储

### 4.7 隐私空间

- 独立入口（侧边栏），默认隐藏内容
- 首次使用需设置 4~16 位隐私空间密码
- 进入后显示隐私空间内的文件/文件夹，与普通空间完全隔离
- 隐私空间内的文件在普通文件列表、搜索结果中均不可见
- 退出隐私空间后需重新验证密码才能进入
- 支持将普通文件移入/移出隐私空间
- 即使管理员也无法通过管理端查看用户隐私空间内容（代码层强制隔离）

### 4.8 分享功能

- 生成分享链接（含可选提取码）
- 分享配置项：
  - 有效期：1天 / 7天 / 30天 / 永久
  - 提取码：自动生成 4 位码，或自定义，或不设置
  - 最大下载次数：不限 / 1次 / 5次 / 自定义
- 分享链接国内直连可访问（经 Cloudflare Workers）
- 端到端加密下的分享密钥设计：
  - 将 DEK 的解密所需片段嵌入分享 token（ECDH 派生），访客无需登录即可解密
  - 分享取消后，token 失效，历史链接无法再解密文件
- 我的分享：查看所有分享记录，支持一键取消分享
- 分享被访问时，文件所有者收到通知（可在设置中关闭）

### 4.9 搜索与排序

**搜索：**
- 全局搜索（文件名模糊匹配）
- 筛选维度：类型、时间范围（开始/结束日期）、大小范围、标签
- 搜索历史记录（最近 20 条，可清空）
- 结果高亮匹配关键词

**排序：**
- 文件名（A-Z / Z-A）
- 上传时间（新→旧 / 旧→新）
- 文件大小（大→小 / 小→大）
- 类型分组

## 五、存储与传输技术设计

### 5.1 分片上传流程

```
前端                          Workers                       Telegram
 │                              │                              │
 │─── 加密文件分片 0..n ───────►│                              │
 │                              │─── sendDocument(chunk_i) ──►│
 │                              │◄── { file_id } ──────────────│
 │◄── { chunk_file_id } ────────│                              │
 │                              │                              │
 │─── 上传完成通知(all file_ids)──► 后端数据库                   │
```

- Telegram 每条消息最大 2GB，超出自动拆分为多条消息
- 下载时后端按 file_id 列表顺序下载各分片并流式拼接返回
- 分片元数据：`{ index, file_id, size, checksum }` 存入数据库

### 5.2 端到端加密技术细节

```
加密流程：
  plaintext ──► AES-256-GCM(DEK) ──► ciphertext ──► 上传 Telegram
  DEK ──► AES-256-GCM(MEK) ──► encrypted_DEK ──► 存入数据库
  MEK = PBKDF2(用户密码, 盐, 310000次迭代, SHA-256)  [不存服务端]

解密流程：
  用户登录 ──► 派生 MEK ──► 解密 encrypted_DEK ──► 得到 DEK
  下载 ciphertext ──► AES-256-GCM 解密(DEK) ──► plaintext
```

### 5.3 元数据数据库核心表设计

```sql
-- 用户表
users: id, username, password_hash, email_encrypted, phone_encrypted,
       role, quota_bytes, used_bytes, mek_salt, created_at, status

-- 文件/文件夹统一节点表
nodes: id, user_id, parent_id, name, type(file|folder),
       size, mime_type, md5_plain,
       is_locked, lock_hash,
       is_private,
       created_at, updated_at, deleted_at

-- 文件分片表
file_chunks: id, node_id, chunk_index, tg_file_id, size, checksum

-- 文件加密密钥表（与 nodes 1:1）
node_keys: id, node_id, encrypted_dek, iv, salt

-- 分享表
shares: id, node_id, user_id, token, password_hash,
        expire_at, max_downloads, download_count,
        share_key_fragment, created_at

-- 标签
tags: id, user_id, name
node_tags: node_id, tag_id

-- 审计日志
audit_logs: id, user_id, action, node_id, ip, ua, created_at
```

## 六、安全体系

### 6.1 数据加密总览

| 层级 | 方案 |
|------|------|
| 传输层 | HTTPS/TLS 1.3（Cloudflare 自动提供） |
| 文件层 | AES-256-GCM 端到端加密，客户端加解密，服务端只存密文 |
| 密钥层 | DEK 由 MEK 加密后存库；MEK 由用户密码 PBKDF2 派生，不存服务端 |
| 密码层 | bcrypt（cost=12）哈希存储所有密码 |
| 配置层 | Bot Token、Channel ID 存于加密环境变量，禁止明文入代码 |
| 数据库层 | 用户手机号、邮箱 AES-256 加密存储 |

### 6.2 认证与授权

- JWT（Access Token，2 小时有效）+ Refresh Token（30 天，HttpOnly Cookie）
- RBAC 权限模型：admin / user 两种角色
- API 接口强制鉴权，所有数据操作前校验 user_id 归属（防越权）
- 管理员 API 单独路由前缀，双重验证（JWT + 角色校验）

### 6.3 接口安全

| 防护类型 | 方案 |
|----------|------|
| 暴力破解 | 登录/验证码接口限流（IP 维度：每分钟 10 次） |
| SQL 注入 | ORM 参数化查询，禁止拼接 SQL |
| XSS | 输入 HTML 转义 + CSP 响应头 |
| CSRF | 非 GET 请求校验 Origin 头 |
| 越权访问 | 所有查询强制附加 `WHERE user_id = ?` |
| 文件安全 | 文件存 Telegram，服务器不执行上传文件 |

### 6.4 Bot Token 安全策略

- Token 仅存于 Cloudflare Workers Secrets，不打印、不写日志
- 定期轮换建议：每 90 天或发现泄露时立即到 @BotFather 重新生成
- 轮换步骤：BotFather 生成新 Token → 更新 Workers Secret → 验证功能正常

### 6.5 隐私空间安全

- 隐私空间密码独立 bcrypt 哈希存储
- 数据库查询层强制过滤 `is_private=true` 的节点（非隐私空间会话不可见）
- 管理员管理端代码层禁止查询用户隐私空间内容

## 七、管理端功能

### 7.1 用户管理

- 用户列表：搜索（用户名/手机号/邮箱）、分页、排序
- 创建用户（管理员直接创建，跳过验证码）

## 八、性能与可靠性

### 8.1 性能指标

| 指标 | 目标 |
|------|------|
| 页面首屏加载 | < 2s（CDN 加速静态资源） |
| 文件列表接口响应 | < 300ms（Redis 缓存） |
| 上传速度 | 充分利用用户上行带宽（并发 3 分片） |
| 下载速度 | 充分利用 Cloudflare 节点带宽 |
| 并发用户 | 支持 1000+ 同时在线 |
| Telegram API 成功率 | > 99.5% |

### 8.2 弱网适配

- 断点续传：记录已上传分片 index，刷新页面后自动续传
- 自动重试：分片失败后指数退避重试（1s → 2s → 4s，最多 5 次）
- 弱网降级：检测到高延迟时并发分片数自动从 3 降至 1
- 上传队列持久化：存入 IndexedDB，页面刷新后恢复
- 单片超时：上传超过 30s 自动触发重试
- 离线感知：网络断开时暂停所有任务，恢复后自动续传

### 8.3 缓存策略

- 文件列表：Redis 缓存（TTL 30s，文件操作后主动失效）
- 下载外链：Workers KV 缓存（TTL 与签名有效期一致）
- 缩略图：浏览器 Cache-Control max-age=86400

### 8.4 高可用

- 后端服务无状态，支持水平扩展（Docker 多副本 + Nginx 负载均衡）
- 数据库主从备份（读写分离）
- Cloudflare Workers 全球冗余，自动故障转移
- 关键操作幂等设计（分片上传、创建文件夹附带幂等 key，防重复提交）

### 8.5 容灾与备份

- 数据库每日自动备份，保留 30 天
- file_chunks 表（存储 Telegram file_id）为最高优先级备份对象
- 监控告警：服务异常 < 1 分钟内邮件通知
